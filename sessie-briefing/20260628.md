## Creator.js v2.2.0 — GPX verbeteringen (toegevoegd na sessie 06)

### gpx_raw export/import
- JSON export bevat nu `gpx_raw`: de volledige GPX-tekst als string
- Bij JSON import: als `gpx_raw` aanwezig → GPX volledig hersteld via `parseGpx()`, trackpunten en kaart werken
- Zonder `gpx_raw` (oudere exports): fallback naar `gpx_stats` read-only (achterwaarts compatibel)
- Status badge toont `✓ Uit JSON (GPX aanwezig)` vs `✓ Uit JSON`

### GPS-ruis filtering in parseGpx()
**Stille filters (altijd actief):**
- Stijging/daling: eleDiff < ±2m genegeerd → filtert GPS-hoogte-ruis (was: +423m, correct: ~46m voor Kalmthout)
- Snelheid: eerste 10 trackpunten overgeslagen → filtert GPS koude-start (was: 37.8 km/u, correct: ~8 km/u max)

**Waarschuwing bij twijfel:**
- Snelheidspieken ≥ 3× het gemiddelde worden gedetecteerd
- Niet-blokkerende gele waarschuwing onder GPX-stats met gevonden piekwaarde
- Knoppen: "Negeren" (gefilterde waarde bewaren) of "Toch bewaren" (ruwe waarde terugzetten)
- Geen koppeling aan vervoersmiddel — puur statistisch

### Meerdere segmenten per route (creator.js v2.3.0)
De vroegere enkelvoudige GPX + datum/locatie stap is vervangen door een herhaalbare segmenten-lijst. Elk segment heeft een eigen vervoersmiddel, GPX-upload, datum/locatie/land/regio/plaats en weerdata. Knop "+ Segment toevoegen" voegt een nieuw blok toe; vanaf het tweede segment kan je het verwijderen.

- `state.segments` array, elk segment met eigen `gpx`, `gpxRaw`, `date`, `location`, `weather`, `transport`
- Export: nieuw veld `segments` in de route JSON; root-level velden (`gpx_stats`, `gpx_raw`, `weather`, `location`, `transport`) blijven gevuld vanuit het eerste segment voor achterwaartse compatibiliteit
- Import: herkent zowel de nieuwe `segments` array als oudere JSON zonder segmenten (wordt dan als één segment ingeladen)
- Kaart (creator preview én `route.js` v2.1.0): elk segment wordt getekend als eigen polyline in de kleur van zijn vervoersmiddel, met popup-label op de startmarker. Routes zonder `segments` array vallen terug op de oude enkelvoudige kaart-tekening

### Hike/Trail vervoersmiddel + moeilijkheidsschaal per vervoersmiddel (creator.js v2.4.0)
Naast "Wandelen" is er nu een apart vervoersmiddel "Hike / Trail" voor bergwandelen/zwaardere trails. Elk vervoersmiddel heeft een eigen, passende moeilijkheidsschaal in plaats van één vaste SAC-schaal voor alles:

| Vervoersmiddel | Schaal | Berekeningsbasis |
|---|---|---|
| Walking | W1-W3 | Stijging per km (vlakke wandelschaal) |
| Hike/Trail | T1-T6 (SAC) | Afstand + stijging + weer (oude logica, nu specifiek voor bergwandelen) |
| Cycling | C1-C4 | Klimintensiteit (m/km) + bochtigheid (scherpe bochten/km uit trackpunten) |
| Motorcycle | M1-M4 | Klimintensiteit + bochtigheid + handmatige "kasseien/onverhard" override (tilt minimaal naar niveau 2) |
| Car | A1-A4 | Zelfde als motorcycle |
| Train/Bus/Boat/Plane | — | Geen schaal — geen fysieke/rij-inspanning van toepassing |

Bochtigheid wordt berekend via bearing/hoekverschil tussen opeenvolgende trackpunten (richtingsverandering > 30° = scherpe bocht), zonder nieuwe afhankelijkheden. Automatische berekening kan altijd handmatig overschreven worden per segment; bij vervoerswissel of nieuwe GPX-upload wordt opnieuw automatisch berekend tenzij de gebruiker zelf al koos.

### Bugfixes (creator.js v2.4.1 + v2.4.2)
- `track_points` ontbrak in `segments[].gpx_stats` en root-level `gpx_stats` export — opgelost in v2.4.1
- `gpx_raw` fallback in route.js v2.2.0: ontbrekende `track_points` worden client-side herparst uit `gpx_raw`
- `stats.maxSpeed` i18n sleutel ontbrak — fallback in route.js + toegevoegd aan beide i18n bestanden
- Open-Meteo weerdata: datum-in-toekomst validatie vóór API-aanroep (v2.4.2), `resp.ok` check, `data.daily` aanwezigheidscheck

### Segmenten zichtbaar op route detail pagina (route.js v2.3.0)
Per segment een compacte blok met gekleurde header (vervoersmiddel + label + nummer), gevolgd door een tweekoloms tabel: GPX-stats + moeilijkheidsgraad links, weerdata rechts. Weerdata gebruikt het startpunt van elk segment als referentie. Op mobiel schakelt naar één kolom. Enkel zichtbaar als `route.segments` aanwezig is.

## Open bug — Footer i18n (ONOPGELOST)

**Probleem:** Op `route.html` en `wandelingen.html` toont de footer `© common.footer.copyright` in plaats van `© MyTrailWalks`.

**Vaststellingen:**
- Werkt correct op `index.html` en `construction.html`
- `i18next.isInitialized` → `true`
- `i18next.t("common:footer.copyright")` → `"MyTrailWalks"` ✓
- `i18next.getResourceBundle("nl", "common")` → correct object ✓
- Manueel `textContent` instellen werkt ✓
- `applyTranslations()` in console geeft `undefined` maar werkt niet
- Footer span bevat de sleutel als tekst: `<span data-i18n="common:footer.copyright">common.footer.copyright</span>`
- Hard refresh heeft geen effect
- Meerdere fixes geprobeerd in `app.js` zonder succes

**Waarschijnlijke oorzaak:** Iets schrijft de sleutel als tekst in de span NADAT `applyTranslations()` al uitgevoerd is. Nog te onderzoeken of `injectComponent` meer dan één keer aangeroepen wordt.

**Bestanden gewijzigd zonder succes:** `app.js` v3.1.0 → v3.2.0 → v3.3.0
